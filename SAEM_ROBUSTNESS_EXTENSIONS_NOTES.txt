SAEM robustness extensions: saemix reference vs NoLimits

Scope
- Focus only on four SAEM-loop robustness features:
  1. SA schedule
  2. Simulated annealing
  3. Multi-chain stabilization
  4. Proposal adaptation
- FIM behavior is intentionally out of scope for this note.
- Constraint for NoLimits: preserve support for many different random-effect and outcome distributions. Do not assume Gaussian random effects or closed-form variance updates globally.


1. SA schedule

saemix: what exists and where
- Control surface:
  - ../saemix/R/SaemixObject.R:335-355
- Runtime setup:
  - ../saemix/R/main_initialiseMainAlgo.R:281-289
- Main loop:
  - ../saemix/R/main.R:184-246
- Current behavior:
  - Total SAEM iterations are split into two phases via nbiter.saemix = c(K1, K2).
  - The stochastic approximation step size is:
    - 0 for the first nbiter.burn iterations
    - 1 during phase 1
    - 1 / k during phase 2
  - There is also a separate nbiter.sa parameter controlling how long simulated annealing remains active.

NoLimits: current behavior and where
- Control surface:
  - src/estimation/saem.jl:260-300
  - docs/src/estimation/saem.md:85-92
- Gain function:
  - src/estimation/saem.jl:355-358
- Main loop use:
  - src/estimation/saem.jl:2191-2193
- Current behavior:
  - The gain is Robbins-Monro style:
    - gamma_t = 1 for t <= t0
    - gamma_t = (t - t0)^(-kappa) for t > t0
  - There is no explicit zero-step burn-in phase.
  - There is no built-in two-phase deterministic schedule matching saemix.

Gap relative to saemix
- NoLimits already has a valid SA schedule, but it does not expose the more hand-stabilized two-phase + burn-in policy used in saemix.
- This is not a flexibility problem. It is a missing schedule mode.

What needs to change in NoLimits
- Add an explicit SA schedule mode to the SAEM constructor and SAEMOptions.
- Recommended API shape:
  - sa_schedule = :robbins_monro | :two_phase | :custom
  - sa_burnin_iters::Int = 0
  - sa_phase_iters::Tuple{Int, Int} or separate sa_phase1_iters / sa_phase2_iters
  - sa_schedule_fn = nothing for fully custom gamma logic
- Keep the current t0/kappa path as the default to avoid breaking behavior.
- Implement the schedule selection in or around:
  - src/estimation/saem.jl:355-358
  - src/estimation/saem.jl:2191-2193
- Extend diagnostics so each iteration records:
  - the active schedule mode
  - the effective gamma
  - whether the iteration is in burn-in / phase 1 / phase 2


2. Simulated annealing

saemix: what exists and where
- Control surface:
  - ../saemix/R/SaemixObject.R:335-355
- Runtime setup:
  - ../saemix/R/main_initialiseMainAlgo.R:285-289
- Application in M-step:
  - ../saemix/R/main_mstep.R:84-132
- Current behavior:
  - During early iterations, saemix applies simulated annealing to:
    - the diagonal random-effect variances
    - the residual error parameters
  - The annealing is coupled to nbiter.sa and alpha.sa.
  - This is a robustness device to prevent variance / residual scales from collapsing too aggressively too early.

NoLimits: current behavior and where
- Existing annealing feature:
  - src/estimation/saem.jl:101-148
  - src/estimation/saem.jl:2202-2211
  - docs/src/estimation/saem.md:252-307
- Current behavior:
  - NoLimits supports anneal_to_fixed only.
  - That feature progressively shrinks selected Normal random-effect prior SDs to a near-zero value.
  - It is a structural feature for collapsing chosen REs into fixed effects.
  - It is not a general SAEM robustness annealing mechanism for covariance or residual scale parameters.

Gap relative to saemix
- NoLimits does not currently have a general early-iteration stabilization layer for:
  - RE covariance / dispersion parameters
  - residual scale parameters
- The existing anneal_to_fixed is much narrower:
  - it requires Normal REs
  - it requires literal SDs
  - it changes the latent prior, not the generic M-step stabilization policy

What needs to change in NoLimits
- Do not copy saemix literally as "anneal omega diagonal", because NoLimits must stay distribution-generic.
- Instead add a generic parameter-annealing framework that can cover Gaussian and non-Gaussian models when appropriate.
- Recommended design:
  - Keep anneal_to_fixed unchanged as a separate feature.
  - Add a new SAEM option family for parameter stabilization, for example:
    - sa_anneal_targets
    - sa_anneal_schedule
    - sa_anneal_strength or sa_anneal_alpha
    - sa_anneal_iters
    - sa_anneal_mode = :floor | :blend | :custom
- The implementation should operate on named fixed-effect targets or block targets, not on hard-coded omega assumptions.
- Built-in support can be provided for common cases:
  - Gaussian RE covariance targets exposed through re_cov_params / builtin_stats
  - residual scale targets exposed through resid_var_param
- For broader distribution support, add a callback-based path:
  - a user hook that receives the current iteration, current parameter values, and candidate updates
  - returns annealed / stabilized updates for selected targets
- Likely insertion points:
  - after built-in closed-form updates are assembled:
    - src/estimation/saem.jl:2247-2273
  - after built-in mean updates are assembled:
    - src/estimation/saem.jl:2278-2318
  - before constants are clamped and the free-set is rebuilt:
    - src/estimation/saem.jl:2318-2331
- Important constraint:
  - for fully numeric M-steps, annealing may need to be expressed as either:
    - temporary post-update projection on named targets
    - temporary penalties / regularization during the M-step
  - this part cannot be handled by Gaussian closed-form formulas alone


3. Multi-chain stabilization

saemix: what exists and where
- Control surface:
  - ../saemix/R/SaemixObject.R:335-355
- Small-N auto-increase of chains:
  - ../saemix/R/SaemixObject.R:181-185
- Replicated-chain data object:
  - ../saemix/R/SaemixData.R:315-327
- Use in initialization:
  - ../saemix/R/main_initialiseMainAlgo.R:180-189
- Use in main algorithm:
  - ../saemix/R/main.R:173-184
  - ../saemix/R/main_mstep.R:24-34
- Current behavior:
  - saemix can run multiple chains for the latent simulation step.
  - If N is small, it may automatically increase the number of chains.
  - The M-step statistics are averaged across chains before stochastic approximation updates.

NoLimits: current behavior and where
- Single-state per batch setup:
  - src/estimation/saem.jl:2155-2163
- E-step refresh for selected batches:
  - src/estimation/saem.jl:2213-2234
- SA update from current latent state:
  - src/estimation/saem.jl:2238-2243
- Current behavior:
  - NoLimits has one current latent vector per batch.
  - Parallelism is through threads / batches, not replicated SAEM chains.
  - update_schedule can subsample batches, but that is not the same as replicated chains.

Gap relative to saemix
- No replicated SAEM chains.
- No small-N heuristic such as "increase chains when subject count is low".
- No averaging of sufficient statistics across parallel latent chains.

What needs to change in NoLimits
- Add chain replication as an explicit SAEM feature, not as a threading side effect.
- Recommended API shape:
  - n_chains::Int = 1
  - auto_small_n_chains::Bool = false
  - small_n_chain_target::Int = 50 or similar heuristic control
- The following state will need a chain dimension:
  - last_params
  - b_current
  - sample_store, or an equivalent chain-aware latent snapshot store
  - optional diagnostics
- The E-step must sample each updated batch independently for each chain.
- The SA update must average chain contributions before applying gamma.
- Likely code regions to change:
  - initialization:
    - src/estimation/saem.jl:2155-2163
  - E-step update loop:
    - src/estimation/saem.jl:2213-2234
  - sufficient-statistics update path:
    - src/estimation/saem.jl:2238-2243
  - custom suffstats callback contract:
    - docs/src/estimation/saem.md:206-213
- Important constraint for distribution generality:
  - chain replication is distribution-agnostic if each chain simply reuses the existing batch posterior sampler
  - no Gaussian-specific logic is needed here


4. Proposal adaptation

saemix: what exists and where
- Control surface:
  - ../saemix/R/SaemixObject.R:335-355
- Initialization of proposal scales:
  - ../saemix/R/main_initialiseMainAlgo.R:277-289
- Adaptive updates:
  - ../saemix/R/main_estep.R:39-95
- Current behavior:
  - saemix maintains random-walk proposal scales domega2.
  - Kernels 2 and 3 adapt these scales toward a target acceptance rate proba.mcmc.
  - This is a direct SAEM-specific proposal adaptation mechanism.

NoLimits: current behavior and where
- SAEM E-step batch sampler call:
  - src/estimation/saem.jl:2217-2231
- Batch sampling implementation:
  - src/estimation/mcem.jl:462-470
- Current behavior:
  - NoLimits delegates the E-step to the provided sampler, currently through the Turing-based batch path.
  - warm_start is supported, but there is no explicit SAEM-level proposal adaptation state.
  - There is no target-acceptance controller analogous to saemix.

Gap relative to saemix
- No SAEM-specific adaptive proposal scaling.
- No acceptance-history-based proposal state kept across iterations.

What needs to change in NoLimits
- This feature must be designed carefully to preserve support for many different distributions.
- Recommended direction:
  - introduce a SAEM sampler abstraction instead of assuming every sampler behaves like a Turing one-shot call
  - support at least two classes:
    - generic external sampler path (current behavior)
    - internal adaptive random-walk MH path for latent batch vectors
- Proposal adaptation should then be enabled only for samplers that expose a proposal scale.
- For samplers such as NUTS / HMC:
  - keep the current behavior
  - mark SAEM-specific proposal adaptation as unsupported or no-op
- For an internal RW-MH path:
  - maintain per-batch, and eventually per-batch-per-chain, proposal scales
  - record acceptance counts each iteration
  - update scales toward a target acceptance rate
- Likely code regions to change:
  - sampler state initialization near:
    - src/estimation/saem.jl:2155-2163
  - E-step call site:
    - src/estimation/saem.jl:2217-2231
  - batch sampler implementation:
    - src/estimation/mcem.jl:462-470
- Important constraint for distribution generality:
  - the adaptive kernel should target the generic latent batch posterior, not a Gaussian-specific closed form
  - proposal adaptation must be expressed in the latent coordinate system already used by the batch sampler
  - if some latent coordinates require constrained support, the implementation will need either:
    - unconstrained internal coordinates, or
    - feature gating to only supported samplers / latent parameterizations


Recommended implementation order

1. SA schedule
- Lowest conceptual risk
- Pure control-flow extension
- Fully distribution-agnostic

2. Multi-chain stabilization
- Medium complexity
- Also fully distribution-agnostic if built on top of the existing batch sampler
- Likely the biggest small-sample robustness gain after scheduling / annealing

3. Proposal adaptation
- Medium to high complexity
- Should probably be tied to a dedicated adaptive RW-MH latent sampler rather than generic Turing samplers

4. Simulated annealing
- High value, but harder to design correctly in a distribution-generic framework
- Best implemented as a new generic parameter-stabilization layer, not as a literal copy of saemix's omega-only logic


Bottom line

- SA schedule: missing schedule mode, easy to add.
- Simulated annealing: currently too narrow in NoLimits; needs a generic target-based stabilization design.
- Multi-chain stabilization: missing; should be added as a true replicated-chain SAEM feature, not confused with threading or minibatching.
- Proposal adaptation: missing; should be added through an explicit adaptive latent-kernel path, not by assuming all Turing samplers can be tuned uniformly.
